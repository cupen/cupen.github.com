聊聊 UID 的实现

啊~ UID，你比 ID 多个 U，   
啊~ UID，你比 UUID 少个 U。   
                                         --无名吹水师

我这几年工作中常常会用到 UID ，虽有不少开源实现，但在不同场合使用哪种方案总免不了计较下。因此我决定写一篇吹水文做点浅薄介绍，以便向人解释我的思路时能更从容些。

UID (Unique identifier) 顾名思义是一种唯一性 ID，但区别于 GUID/UUID ，它只是有限范围内唯一。 比如在游戏场景里，玩家ID、野怪ID、对局ID 等等会用各自的 UID 算法实现，因为有类型做区分，所以类型不一样的对象 ID 重复是无所谓的。

下面的代码展示了一个最原始的 UID 实现（别笑，高级算法无非就是分布式计数器）。

var countor uint32
func NextId() uint32 {
  return atomic.AddUint32(&countor)
}

这个实现只保证进程内唯一，且重启后计数器重置，无法保证唯一。 然而多数场合下，我们需要既能跨进程唯一，且重启后继续保证唯一的算法。

有两种改进思路：
一种是引入一个 borker 用来协调不同进程的 ID 分配范围，并将已分配出去的 ID 区间保存到数据库。就跟分蛋糕一样，你一口我一口，吃一口少一口。

另一种则是在 UID 里加入节点 ID 和时间戳，其中节点 ID 用来保证跨进程唯一，而时间戳则保证计数器重置也不影响唯一性。但在同一个时间戳里，计数器重置还是会破坏唯一性，所以算法会确保单位时间内只能生成一定数量（由计数器最大值决定）以杜绝这种情况。
以下是实现参考。

  • Snowflake
    最早的实现，由 Twitter 提出，是为全局有序（k-sorted）开发的 int64 UID 算法。
    https://blog.twitter.com/engineering/en_us/a/2010/announcing-snowflake
    文中给出了 Scala实现， 也可参考 Wiki说明 :
     
    1 bit 保留
    41 bits 相对时间戳（毫秒），2^41 毫秒大约是 69 年，这之后会溢出导致重复。
    10 bits 节点 ID，最多 1024 个节点。
    12 bits 计数器。
    算法性能每毫秒 2^12 个 UID，大约是 409w/sec.

  • Sonyflake
    索尼大法加持后的 snowflake ，降低性能，提升易用性和使用寿命。
    https://github.com/sony/sonyflake
     
    39 bits 相对时间戳（以10毫秒为单位），相当于 174年。
    8 bits 计数器。
    16 bits 节点 ID。
    算法性能每 10 毫秒 2^8 个 UID，大约是 2.5w/sec.
    = =! 但我实测跑出了 5w 的 QPS…… 回头有空再跟吧……
    https://github.com/sony/sonyflake/pull/9

  • MongoDB ObjectId
    这个实现长度在 12 字节，实际使用时可以用 hex string。
    https://www.mongodb.com/docs/manual/reference/method/ObjectId/
     
    4 字节 unix 绝对时间戳（秒） + 5 字节随机数 + 3 字节计数器
    算法性能每秒 2^24 个 UID，就是 1600w/sec.
    但由于随机数的缘故，在多进程环境下可能会有重复，理论上是 (进程数-1) / (2^40) 的概率，大约万亿分之几吧。
     
    另外，该算法还有一种实现。
    参考 mgo(mongodb driver) 的旧版 ObjectId 算法。
    其与最新算法不同之处在于 5 字节随机数改由 3 字节随机数 + 2 字节进程 ID，这可以保证同一台主机里多进程生成的 ID 绝对唯一。但在容器环境下，由于进程 ID 可能会重复，结果可能更杯具。
    https://github.com/globalsign/mgo/blob/eeefdecb41b842af6dc652aaea4026e8403e62df/bson/bson.go#L266-L283
     
    当然，这个随机数带来的风险可以通过人工指定节点 ID 来摆平。

综上，如果是游戏服务端，使用 Snowflake 的思路是不错的。

  1. 节点数并不多。
  2. 性能要求中等。
  3. 运营时间短，如暴雪这般成功也不过 20 年。

只要适当调整下时间戳、节点 ID 、计数器的 bit 长度，控制在 64bit 是可以的.
至于时间戳长度有限的问题，百年后留给后人解决吧，大不了整一套 128bit 的软硬件。
